# v0.5.9 릴리즈 노트

v0.5.8 이후 변경 사항입니다.

## 사용자-visible 변경

- `namba queue start/resume`에 queue runner 선택 경로를 추가했습니다. CLI/headless 환경은 기존 `codex exec` 기반 실행을 유지하고, Codex Desktop 환경은 nested `codex exec` 없이 request/evidence handoff 상태로 멈추도록 했습니다. (PR #65, d244bbf)
- Desktop runner handoff는 `.namba/logs/runs/<spec>-request.json`, `.md`, heartbeat, 공통 queue evidence schema를 사용해 현재 Codex Desktop 세션이 구현/검증한 결과를 다음 `namba queue resume`에서 이어받을 수 있습니다. (PR #65, d244bbf)
- `namba queue doctor`와 `namba queue recover`를 추가해 stale `running` 상태를 명확한 blocker로 복구할 수 있게 했습니다. (PR #65, d244bbf)

## 수정

- Desktop handoff 후 새 커밋이 생긴 경우에도 queue가 이전 `LastObservedHeadSHA`로 evidence를 거절하지 않도록 resume 시 현재 HEAD를 갱신합니다. (6316803)
- post-run checkpoint(`checks_pending` 등)를 runner stale로 오판하지 않도록 stale recovery를 실제 run phase에만 적용합니다. (8deed98)
- CLI runner가 completed evidence를 남겼지만 queue state가 pre-run SHA에 머문 crash/restart 구간에서도 stale 판정 전에 HEAD를 갱신해 정상 진행하도록 했습니다. (eda952d)
- queue-owned runtime 파일 때문에 branch clean gate나 queue PR staging이 실패하지 않도록 `.namba/logs/queue/` 및 `.namba/logs/runs/`를 런타임 경로로 취급합니다. (PR #65, d244bbf)

## 내부 유지보수

- CLI runner 실행 로그에 command args, stdout/stderr artifact, exit code를 기록하고, Codex CLI가 지원하면 structured `--json` 출력을 사용합니다. (PR #65, d244bbf)
- queue runner 공통 evidence/heartbeat fixture와 stale recovery 회귀 테스트를 보강했습니다. (PR #65, 6316803, 8deed98, eda952d)
